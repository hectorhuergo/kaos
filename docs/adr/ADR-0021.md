# ADR-0021 — GitHub Copilot Provider (device-flow OAuth)

Status: Accepted

## Context

KAOS es *AI Provider Agnostic*: los agentes piden completions a través del
contrato `LLMProvider` y cualquier backend se enchufa detrás. Ya existían
`openai`, `github` (GitHub **Models**), `anthropic`, `ollama` y `echo`.

Faltaba usar la **GitHub Copilot API** (`https://api.githubcopilot.com`), que es
distinta de GitHub Models: requiere una **suscripción Copilot** y una cadena de
autenticación en dos pasos que ningún otro provider tenía:

1. Un **token OAuth de GitHub** (`gho_…`) de larga duración, obtenido con el
   *device flow* (sin pedirle al usuario que genere un PAT a mano).
2. Ese token se **intercambia** por un **token de sesión efímero** de Copilot
   (`GET api.github.com/copilot_internal/v2/token`, expira ~30 min), que se usa
   como `Bearer` junto con headers de editor (`Editor-Version`,
   `Copilot-Integration-Id`, `Openai-Intent`, …).

## Decision

### A. Nuevo provider `copilot`

- `LLM_PROVIDERS` incorpora `copilot`; `Settings.copilot_oauth_token` se lee de
  `KAOS_COPILOT_TOKEN` (o `GITHUB_COPILOT_TOKEN`). El catálogo
  (`core/providers.py`) lo describe (endpoint, modelo por defecto `gpt-4o`,
  secreto propio, sin fallback cruzado a otros tokens).
- `CopilotLLMProvider` **extiende** `OpenAICompatibleLLMProvider`: la API de chat
  es OpenAI-compatible, así que se reutiliza el request + los reintentos por
  rate-limit. Se agregan dos puntos de extensión mínimos al padre:
  `extra_headers` (headers fijos) y un hook `async _auth_token()` que el hijo
  usa para **acuñar y cachear** el token de sesión, refrescándolo poco antes de
  expirar (`expires_at` con un margen de seguridad).
- La composición root (`bootstrap/factory.build_llm`) construye el provider a
  partir del token OAuth. El modelo configurado se respeta tal cual (el default
  de la app, `gpt-4o-mini`, también es un modelo válido de Copilot).

### B. Autologin por *device flow*

- `plugins/providers/copilot.py` expone `request_device_code`,
  `poll_for_access_token` y `device_login(on_prompt=…)`. El polling maneja
  `authorization_pending`, `slow_down`, `expired_token` y `access_denied`, y es
  testeable inyectando `client` y `sleep`.
- `kaos copilot login` corre el flujo: imprime la URL + el código, espera la
  autorización y **persiste** el token en el credential store (si hay
  `KAOS_DATABASE_URL`) o lo imprime para `.env`. `kaos copilot status` verifica
  que el token pueda generar una sesión.

## Consequences

- Copilot queda disponible en CLI y en la consola web (aparece en el catálogo de
  providers como cualquier otro). El secreto es *write-only* desde la API.
- El `client_id` del device flow es público (viaja en los clientes oficiales del
  editor); no es un secreto y se mantiene como constante.
- Se mantiene la compatibilidad hacia atrás: ningún provider existente cambia su
  comportamiento; el refactor de `OpenAICompatibleLLMProvider` es aditivo.
