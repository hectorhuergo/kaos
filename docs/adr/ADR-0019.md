# ADR-0019 — Cross-Workspace Relations (Project Graph)

Status: Accepted

## Context

El Knowledge Graph (ADR-0010) proyecta el conocimiento como nodos `workspace`,
`artifact` y `event`, con aristas `contains` (workspace → artifact) y
`derived_from` (artifact → event). Todas esas aristas viven **dentro de un mismo
workspace**, por lo que el grafo renderizado muestra una **isla por workspace**:
cada foro/canal/repo es un componente conexo con forma de estrella, sin ninguna
relación con los demás.

En la práctica un mismo **proyecto** se reparte entre varios workspaces. Ejemplo
real del dogfooding: el foro de Discord `proyecto-x` y los repos de GitHub
`proyecto-x-grid` y `kaos`. El conocimiento de `proyecto-x` (foro) y
`proyecto-x-grid` (repo) pertenece al mismo esfuerzo, pero el grafo los mostraba
como islas separadas, perdiendo la lectura "todo lo que sabemos del proyecto".

## Decision

Introducir una primera capa de **relaciones entre workspaces**, determinista y
sin LLM ni red, con **tres señales complementarias**: agrupamiento explícito por
proyecto (autoritativo), relaciones ad-hoc entre suscripciones, y una heurística
basada en el **nombre del proyecto**.

### Modelo

- Nueva arista `related_to` (workspace ↔ workspace) en `kaos.core.knowledge`.
  Es no dirigida; se emite **una** arista por par, con los ids en orden estable
  (ordenado), **deduplicada** entre las señales y restringida a los workspaces en
  vista.
- Función pura
  `relate_workspaces(labels, *, projects=None, relations=None, min_shared_tokens=2)`:
  - Recibe los **labels ya resueltos** (nombre amigable por workspace) y, de forma
    opcional, un mapa `projects` (workspace → proyecto) y `relations` (workspace →
    workspaces relacionados), manteniendo el Core agnóstico de cómo se obtienen.
  - **(1) Agrupamiento explícito (autoritativo):** dos workspaces con el mismo
    `project` (case-insensitive) se relacionan, **aunque sus nombres no compartan
    nada**. Así el repo `kaos` —el *cerebro* de `proyecto-x`— se une al proyecto.
  - **(2) Relaciones ad-hoc:** pares que un operador conectó a mano vía
    `related_to`, aunque no compartan nombre ni proyecto.
  - **(3) Heurística de nombre:** toma el segmento más específico del nombre (para
    GitHub `owner/repo` usa `repo`), lo tokeniza, y relaciona si comparten al menos
    `min_shared_tokens` tokens iniciales: `proyecto-x` y `proyecto-x-grid`.

### Persistencia y edición

- `Subscription` gana un campo opcional `project: str | None`. Persiste en
  `kaos_subscriptions` (columna `project`, con migración
  `ADD COLUMN IF NOT EXISTS`). Se puede definir al crear la suscripción:
  - CLI: `kaos subscribe <id|owner/repo> [--github] --project proyecto-x`
  - Consola web: campo **Proyecto (opcional)** en el formulario de suscripción.
- `Subscription` también gana:
  - `related_to: list[str]` — relaciones **ad-hoc** a otros workspaces (columna
    `related_to text[]`). Genera aristas `related_to` aunque no compartan nombre
    ni proyecto. CLI: `--related ws1,ws2`; consola: multiselección "Relacionada
    con".
  - `publish_default: bool = True` — si una corrida **automática** (scheduler /
    `kaos run`) publica en Discord esta suscripción (columna `publish_default`,
    default `true` para preservar el comportamiento histórico). En `False` la
    suscripción sigue leyéndose y **persistiéndose** (conocimiento), pero se
    imprime en consola en vez de publicarse. La corrida **interactiva** de la
    consola mantiene su propio toggle explícito de publicación.
- Edición: `PATCH /api/subscriptions/{channel_id}` actualiza parcialmente
  `project`, `publish_default`, `related_to`, `resume_thread_id` e
  `interval_seconds`; la consola expone un panel "Editar" por suscripción.
- El grafo se construye con `project` y `related_to` de cada suscripción activa
  (`service.load_projects` / `service.load_relations`) y se pasa a
  `relate_workspaces` como los mapas `projects` y `relations`.

### Aplicación (capa de vistas)

- La resolución de nombres amigables (`resolve_labels`, vía Discord REST) vive en
  `plugins/dashboard`, por lo que las relaciones se calculan **en la capa de
  vista**, no en el Core:
  - `GET /` y `GET /api/knowledge` (FastAPI) y el CLI `kaos knowledge` /
    `kaos dashboard` aplican `graph.relabel(labels)` y luego
    `graph.edges.extend(relate_workspaces(labels))`.
- El grafo (Mermaid y JSON) muestra las aristas `related_to` conectando las islas
  de un mismo proyecto.

### Bug relacionado corregido

- El nodo `workspace` se etiquetaba con el id crudo (`discord:<id>`). Se añadió
  `KnowledgeGraph.relabel(labels)` para mostrar el nombre del canal/foro/repo
  sin filtrar el lookup al Core.

## Consequences

- El grafo pasa de "una isla por workspace" a un **grafo de proyecto**: los
  workspaces de un mismo esfuerzo se leen como un solo cuerpo de conocimiento.
- Heurística determinista y trazable (sin LLM, sin costo, reproducible en tests).
  Su límite es que depende del **naming**; por eso se complementa con el
  agrupamiento **explícito** por `project`, que conecta workspaces cuyo nombre no
  comparte prefijo (p. ej. `kaos` como cerebro de `proyecto-x`). `min_shared_tokens`
  ajusta la sensibilidad de la heurística.
- Base para evolucionar hacia relaciones **semánticas** (temas, decisiones,
  riesgos y entidades compartidas entre artifacts de distintos workspaces),
  posiblemente asistidas por agentes, sin cambiar el contrato del grafo: solo se
  agregan tipos de arista.
- El Core permanece agnóstico: recibe labels ya resueltos; la resolución de
  nombres sigue en `plugins/dashboard`.
