# ADR-0014 — Dev Agent & Tool-Using Agents

Status: Accepted

## Context

Hasta ahora los agentes de KAOS (p. ej. el Resume Agent) son *productores de
texto*: consumen un `Context` y devuelven un `Artifact`. El roadmap pide un
primer **agente con capacidades dev** que "se sume al equipo": no solo describir,
sino **actuar** sobre el repositorio (leer archivos, buscar en el código, correr
tests/lint) para producir conocimiento verificado.

Esto introduce una capacidad nueva y sensible: **ejecutar acciones**. Necesitamos
que sea (a) agnóstica del proveedor de LLM —incluyendo modelos locales pequeños
vía Ollama—, (b) trazable, y (c) segura por defecto.

## Decision

### Nuevo contrato `Tool`

- `src/kaos/contracts/tool.py` define `Tool` (Protocol): `name`, `description` y
  `async run(args) -> str` (una *observación* de texto). El Core solo depende del
  contrato; las herramientas concretas viven en plugins.

### Toolbox dev confinado

- `src/kaos/plugins/tools/dev_tools.py`: `ReadFileTool`, `ListDirTool`,
  `SearchCodeTool` y `RunCommandTool`, más `default_dev_tools(root)`.
- **Seguridad por diseño**:
  - Confinamiento: toda ruta se resuelve y se rechaza si escapa del `root`
    (sin traversal).
  - `RunCommandTool` ejecuta **sin shell** (`shlex.split` + `create_subprocess_exec`)
    y solo comandos cuyo prefijo de tokens está en una **allowlist** explícita
    (por defecto: `pytest`, `python -m pytest`, `ruff check`, `mypy`,
    `git status|diff|log|show`). Todo lo demás se rechaza.
  - Timeout por comando y truncado de salida.

### `DevAgent`: loop de uso de herramientas

- `src/kaos/plugins/agents/dev_agent.py` implementa el contrato `Agent`. Dada una
  tarea (`context.params["task"]` o un evento `dev.task`), corre un **loop acotado**:
  en cada turno el LLM responde UN objeto JSON —`{"action":"tool",...}` o
  `{"action":"final",...}`—; si es una herramienta, se ejecuta y la observación
  se realimenta; si es final, se concluye. Límite duro de pasos (`max_steps`) y
  una síntesis final forzada si se alcanza el límite.
- Protocolo **JSON-por-turno** en vez de function-calling del proveedor, para
  funcionar con cualquier `LLMProvider`, incluidos modelos locales.
- Produce un `Artifact` `dev.session` con: `task`, `answer` (Markdown), `steps`
  (cada herramienta + args + observación) y un `summary` renderizable en el
  dashboard. `source_events` mantiene la trazabilidad.

### CLI

- `kaos dev "<tarea>" [--repo-root .] [--max-steps N] [--dry-run]`. Usa la
  composición root (`load_settings` → `build_llm`/`build_storage`), corre el
  agente sobre el repo local y persiste la sesión (salvo `--dry-run`), de modo
  que aparece en el knowledge base y el dashboard.

## Consequences

- KAOS suma su primer **teammate activo**: un agente que inspecciona el código y
  produce conocimiento verificado y trazable, ejecutable **100% local** (Ollama)
  sin costo.
- Nuevo contrato `Tool` estable, base para futuras capacidades (escribir archivos,
  abrir PRs, etc.) siempre detrás de contratos y con allowlist.
- Calidad dependiente del modelo: modelos muy pequeños (p. ej. `llama3.2:3b`)
  siguen el protocolo de herramientas de forma poco fiable; para uso real del Dev
  Agent se recomienda un modelo instruible más capaz (p. ej. `qwen2.5:7b`) o un
  proveedor cloud, seleccionable desde la consola web.
- Riesgo asumido y acotado: la ejecución de comandos está restringida a una
  allowlist de solo lectura/análisis y confinada al repo. Ampliar a acciones de
  escritura (aplicar parches, commits) requerirá una nueva decisión con las
  salvaguardas correspondientes.
