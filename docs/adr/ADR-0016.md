# ADR-0016 — Scheduler Execution Plan (per-subscription cadence)

Status: Accepted

## Context

El scheduler Beta (`kaos schedule`, ADR-0009) corría **todas** las suscripciones
activas en un único intervalo global. Surgió la pregunta de diseño: ¿el plan de
ejecución debería vivir en un **gestor separado** que configure aparte la
suscripción y su cadencia, o **con la suscripción**?

Un gestor separado agrega un segundo modelo/almacén de configuración (qué correr
vs. cuándo correrlo), más superficie y más acoplamiento, para un beneficio que
hoy no necesitamos.

## Decision

**El plan vive con la suscripción.** La `Subscription` es el "qué se observa"; le
sumamos el "cada cuánto":

- `Subscription.interval_seconds: int | None` (default `None` = correr en cada
  pasada del scheduler). Persistido en el store (columna `interval_seconds` en
  `kaos_subscriptions`, con `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` para
  compatibilidad hacia atrás).
- Configurable al suscribir: `kaos subscribe … --every SECONDS` y campo "Plan:
  cada N segundos" en la consola web.

**Un solo proceso scheduler, con granularidad de tick:**

- `kaos schedule` (y el servicio `scheduler` de docker-compose, profile `app`)
  hace *tick* a la cadencia base (`--interval` / `KAOS_SCHEDULER_INTERVAL`).
- En cada tick, una función pura `due_subscriptions(subs, now, last_run,
  base_interval)` decide qué suscripciones están **vencidas** (nunca corridas, o
  transcurrido su `interval_seconds`, con fallback al intervalo base) y solo esas
  se ejecutan vía `run_subscriptions(only=due)`. El scheduler mantiene
  `last_run` por `channel_id` en memoria del proceso.
- La cadencia base es la **granularidad mínima**: una suscripción corre a lo sumo
  cada `interval_seconds`, redondeado al tick. No hay estado de planificación
  persistido ni cron.

## Consequences

- Cadencia por suscripción sin un segundo almacén ni un "gestor de planes": menos
  superficie, coherente con *Plugin First* / *Core Agnostic*.
- `due_subscriptions` es pura e inyectable (reloj `clock`), fácil de testear de
  forma determinista.
- Limitaciones asumidas: `last_run` es por proceso (un reinicio dispara una
  corrida; como `run` es idempotente, es seguro). Para cadencias muy finas
  respecto de la base, hay que bajar el tick. Un scheduler distribuido o cron
  externo con estado compartido queda para más adelante (v1.0) si hiciera falta.
