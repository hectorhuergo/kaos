egou# ADR-0015 — GitHub Subscriptions & Subscription Runner Registry

Status: Accepted

## Context

Las suscripciones (ADR-0008) nacieron para Discord: una `Subscription` con
`kind ∈ {forum, channel}` que el scheduler resume periódicamente. El GitHub
Connector (ADR-0012) ya existía como plugin (`kaos github`), pero **no se podía
suscribir** un repositorio: había que correrlo a mano.

Además, el despacho por tipo en `run_subscriptions` era una cadena `if/elif`
acoplada a cada connector concreto (forum → Discord forum, github → run_github…),
lo que contradice *Plugin First*: sumar una fuente obligaba a editar el loop.

## Decision

### GitHub como tipo de suscripción (dato, no lógica en el core)

- `kaos.domain.subscription`: nuevo `kind = "github"` en `KINDS`. Para una
  suscripción GitHub, `channel_id` es el slug `owner/name` y el `workspace`
  canónico es `github:owner/name`.
- Helpers de dominio: `Subscription.workspace_for_github(repo)` y
  `workspace_for_kind(kind, channel_id)` (el dominio no conoce connectors; solo
  namespacea workspaces).
- CLI: `kaos subscribe <owner/repo> --github`. Web console: opción `github` en el
  selector de tipo; el endpoint deriva el workspace por tipo.

### Registro de runners de suscripción

- `kaos.cli.subscriptions.SUBSCRIPTION_RUNNERS: dict[str, SubscriptionRunner]`
  mapea `kind → handler` (`forum → _run_forum`, `channel → _run_channel`,
  `github → _run_github`). `run_subscriptions` resuelve el runner del registro en
  vez de un `if/elif`.
- Cada runner arma su pipeline reutilizando los plugins existentes
  (`run_forum_backfill`, `run_backfill`, `run_github`), que a su vez registran el
  Connector correspondiente + `ResumeAgent` + Publisher en un `KaosRuntime`.

## Consequences

- Suscribir un repo de GitHub es de primera clase (CLI + consola), y el
  scheduler lo corre igual que un foro.
- El loop de suscripciones queda **agnóstico del connector**: sumar una fuente
  nueva es registrar su runner, sin tocar `run_subscriptions` (extensible por
  composición).
- El acoplamiento se concentra en los runners (borde del sistema), no en el
  dominio ni en el core. La lectura general de Git (trabajo de Maxi) se integrará
  como un runner/connector adicional sin cambiar el contrato de suscripción.
