# ADR-0012 — GitHub Connector

Status: Accepted

## Context

La visión de KAOS incluye el **dogfooding**: "KAOS debe terminar administrando su
propio desarrollo". El roadmap Beta contempla un GitHub Connector. Discord fue el
primer Connector; GitHub es el segundo y el que permite a KAOS transformar la
actividad de un repositorio (commits, issues, pull requests) en conocimiento
estructurado y trazable, empezando por su propio repo.

## Decision

### Source + Connector, espejo del patrón Discord

- `src/kaos/plugins/connectors/github_connector.py` define `GitHubItem`
  (commit/issue/pull_request), el contrato `GitHubActivitySource`, un
  `StaticGitHubSource` para tests y el `GitHubConnector`. Este mapea cada item a
  un evento ``message.created`` (workspace ``github:<owner>/<repo>``) y emite
  ``conversation.completed`` al terminar, de modo que el **Resume Agent** existente
  resume toda la ventana de actividad sin cambios.
- `src/kaos/plugins/connectors/github_rest.py` implementa `GitHubRestSource`:
  lee commits y (opcionalmente) issues/PRs vía la REST API de GitHub con httpx,
  normaliza y ordena cronológicamente. Autentica con `KAOS_GITHUB_TOKEN` (el
  mismo token de GitHub Models sirve si tiene acceso de lectura al repo).

### CLI y configuración

- `kaos github [<owner/repo>] [--dry-run] [--limit N] [--no-issues]`. El repo
  puede venir por argumento o por `KAOS_GITHUB_REPO`. Con `--dry-run` imprime por
  consola; si no, publica según la configuración (bot/webhook/consola).
- Reutiliza el pipeline completo (Runtime → ResumeAgent → Publisher) y la
  composición root (`build_llm`, `build_publisher`, `build_storage`).

## Consequences

- KAOS ahora resume su **propio desarrollo**: `kaos github hectorhuergo/kaos`
  produce un "Estado del Proyecto" a partir de los commits reales (dogfooding).
- El evento sigue siendo ``message.created``, así que el conocimiento de GitHub
  convive con el de Discord en el mismo grafo/dashboard, por workspace.
- El connector es agnóstico del transporte (source inyectable) y testeable sin
  red (`StaticGitHubSource` + `httpx.MockTransport`).
- Próximos pasos naturales: paginación e incrementalidad (solo lo nuevo desde el
  último run, con la cache), y disparo por webhooks de GitHub en vez de polling.
