# ADR-0013 — Web Console & Persisted Runtime Config

Status: Accepted

## Context

Hasta ahora KAOS se administra por CLI y el dashboard vivo (ADR-0011) es
**solo lectura**. Para presentar y operar KAOS sin tocar la terminal hace falta
una **consola web** que permita:

1. Ver y seleccionar el **Provider LLM** activo (provider + modelo).
2. Administrar **Subscriptions** (alta + baja) de forma visual.
3. Navegar los **Dashboards** disponibles por workspace.
4. **Configurar credenciales** de cada provider desde la web (no solo por `.env`).

La tensión de diseño es con ADR-0006/ADR-0008 (Config Split): históricamente los
**secretos** (API keys, tokens) vivían solo en el entorno (12-factor). Para poder
configurar un provider de punta a punta desde la consola, se decide permitir
**persistir el secreto de cada provider en PostgreSQL**, manteniendo el entorno
como **fallback**. *Qué* provider/modelo usar tampoco es un secreto: es estado de
configuración que el operador cambia en runtime y debe sobrevivir reinicios —
igual que una `Subscription`.

## Decision

### Nuevo estado de dominio: `RuntimeConfig` + contrato `ConfigStore`

- `src/kaos/domain/runtime_config.py` define `RuntimeConfig` (llm_provider,
  llm_model, updated_at) — una única configuración activa, keyed por
  `SINGLETON`. Nunca contiene secretos.
- `src/kaos/contracts/config_store.py` define el contrato `ConfigStore`
  (`get()`/`set()`), espejo de `Storage` y `SubscriptionStore`. El Core queda
  agnóstico del backend.
- Backends: `InMemoryConfigStore` (`src/kaos/runtime/config_store.py`) para
  tests/offline y `PostgresConfigStore` (tabla `kaos_runtime_config`) para
  persistencia durable. Seleccionados por `build_config_store(settings)` en la
  composición root según `KAOS_DATABASE_URL`.

### Credenciales por provider: `ProviderCredential` + contrato `CredentialStore`

- `src/kaos/domain/provider_credential.py` define `ProviderCredential`
  (provider, api_key, model, base_url, updated_at): el secreto de un provider más
  overrides opcionales de modelo/endpoint.
- `src/kaos/contracts/credential_store.py` define `CredentialStore`
  (`get`/`set`/`delete`/`list`), mismo patrón que los demás stores.
- Backends: `InMemoryCredentialStore` y `PostgresCredentialStore` (tabla
  `kaos_provider_credentials`, upsert por provider), vía
  `build_credential_store(settings)`.
- El secreto es **write-only** desde la API: se envía al servidor pero nunca se
  devuelve; las rutas de lectura solo exponen si hay credencial guardada
  (`stored`) y la readiness combinada (env o Postgres).

### Overlay en la composición root

- `factory.load_settings(base)` superpone dos tipos de estado durable no-ambiental
  sobre las `Settings` del entorno:
  1. la selección activa (provider + modelo) desde el `ConfigStore`;
  2. la credencial del provider activo (api_key + overrides de modelo/base_url)
     desde el `CredentialStore`, mapeada al campo de `Settings` que lee
     `build_llm` (`llm_api_key`/`github_token`/`anthropic_api_key`).
- El entorno es el **fallback**: si no hay credencial persistida, se usa el valor
  de `.env`. Sin `DATABASE_URL` es un no-op (devuelve `base`).
- Los comandos que producen conocimiento (`backfill`, `backfill-forum`,
  `github`, y por ende `run`/`schedule`) resuelven sus settings con
  `await load_settings(...)`.

### Consola sobre la misma app FastAPI

- La app de ADR-0011 crece con una **superficie de administración** en la misma
  app (no un servicio aparte). Página en `GET /console`
  (`src/kaos/plugins/dashboard/console.py`, HTML autocontenido + JS vanilla, sin
  build step) y rutas JSON:
  - `GET  /api/providers` — catálogo + readiness (env **o** Postgres) + `stored`
    por provider + selección persistida.
  - `PUT  /api/config/provider` — persiste provider + modelo (valida contra
    `LLM_PROVIDERS`).
  - `PUT  /api/providers/{id}/credential` — persiste el secreto de un provider;
    `DELETE` lo borra (vuelve a usar `.env`).
  - `GET  /api/subscriptions`, `POST /api/subscriptions` (alta/upsert),
    `DELETE /api/subscriptions/{channel_id}` (baja).
  - `POST /api/preview/subscription` y `POST /api/preview/github` — **dry-run**:
    resumen mostrado en la consola **sin publicar en Discord** (aun con token
    real en el entorno). Reutilizan los pipelines con un `CapturingPublisher`
    que colecta artefactos en vez de enviarlos; sólo se suprime la publicación,
    la lectura de la fuente sí ocurre.
  - Dashboards: reusa `GET /api/workspaces`; cada uno enlaza a `/?workspace=...`.
- Los stores se construyen una vez al startup y se reutilizan entre requests;
  pueden inyectarse (tests) y se cierran en el shutdown (lifespan).

## Consequences

- KAOS se administra por completo desde el navegador: la consola edita **estado
  durable** (config de runtime, credenciales y suscripciones). Los secretos se
  guardan en PostgreSQL y nunca se devuelven por la API (write-only).
- La selección de provider/modelo **y las credenciales** son persistentes en
  PostgreSQL y se aplican a los runs siguientes vía `load_settings`; `.env` queda
  como fallback. Esto flexibiliza ADR-0008: el secreto puede vivir en el entorno
  o en la DB, con la DB teniendo prioridad para el provider activo.
- Nuevos contratos `ConfigStore` y `CredentialStore` estables, alineados con el
  patrón de `Storage`/`SubscriptionStore` (mismos backends in-memory/PostgreSQL).
- Riesgo asumido: el secreto queda en la base de datos en texto (mismo nivel de
  confianza que el DSN). Próximos pasos naturales: cifrado en reposo de las
  credenciales, autenticación/roles para la consola, más ajustes persistidos
  (intervalo del scheduler) y edición de suscripciones (reactivar/editar).
