# ADR-0010 — Knowledge Layer & Graph

Status: Accepted

## Context

Hasta ahora KAOS producía `Artifact`s (resúmenes por hilo, `project.status`
consolidado) y los persistía en `Storage`, pero no había forma de **inspeccionar
el conocimiento acumulado** como un todo. La visión del proyecto es explícita:
"El objetivo no es producir texto, sino construir conocimiento persistente,
trazable y reutilizable; los reportes y dashboards son solo vistas de ese
conocimiento". Faltaba materializar esa capa de vistas para empezar a probar con
casos reales.

Cada `Artifact` ya referencia los eventos que lo produjeron (`source_events`),
de modo que la estructura trazable **ya existe**; solo faltaba proyectarla.

## Decision

### Knowledge Graph como proyección (no un datastore nuevo)

- `src/kaos/core/knowledge.py` define un modelo **puro** (`KnowledgeNode`,
  `KnowledgeEdge`, `KnowledgeGraph`) y un builder `build_graph(storage,
  workspaces, *, include_events=False)` que lee los artifacts (y opcionalmente
  los eventos) vía el contrato `Storage`. No introduce una base de datos de
  grafos: es un **read-model** sobre lo ya persistido (Knowledge before Reports).
- Nodos: `workspace`, `artifact`, `event`. Aristas: `contains`
  (workspace → artifact) y `derived_from` (artifact → event). Los eventos se
  excluyen por defecto porque una conversación real tiene cientos; el grafo a
  nivel artifact es la vista útil.
- Exportadores: `to_dict()` (JSON) y `to_mermaid()` (diagrama). Los ids reales
  (UUID, `discord:123`) se aliasan a handles `nN` para que el Mermaid sea válido.

### Dashboard como vista estática (v1)

- `src/kaos/plugins/dashboard/` renderiza un **HTML autocontenido**: los
  resúmenes por workspace como tarjetas + el grafo de trazabilidad (Mermaid vía
  CDN). Sin servidor ni build: se abre en el navegador. Es presentación, por eso
  vive en `plugins/` y no en el Core.

### CLI

- `kaos knowledge [--workspace W] [--format text|mermaid|json] [--events]`
- `kaos dashboard [--workspace W] [--out archivo.html] [--events]`
- Si se omite `--workspace`, los workspaces se toman de las **suscripciones
  activas**, así `kaos knowledge` muestra todo lo que KAOS gestiona. Un id de
  Discord "pelado" se normaliza a `discord:<id>`.

## Consequences

- KAOS pasa de "generar resúmenes" a "consultar conocimiento": el grafo y el
  dashboard son la primera capa de vistas, lista para casos reales.
- El grafo es agnóstico del backend (funciona con `InMemoryStorage` o
  `PostgresStorage`) y no agrega dependencias al Core.
- El dashboard estático es el primer paso; un servicio vivo (FastAPI, filtrado,
  búsqueda) queda como evolución natural de la Beta.
- Base para el futuro **Knowledge Graph enriquecido**: relaciones semánticas
  entre artifacts (temas, decisiones, riesgos) por encima de la trazabilidad
  estructural actual.
