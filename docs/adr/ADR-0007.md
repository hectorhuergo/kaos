# ADR-0007 — Summary Cache & Knowledge Reuse

Status: Accepted

## Context

El backfill de un foro resume cada hilo con el LLM. Volver a correrlo sobre los
mismos hilos repetía consultas idénticas al proveedor (lento y limitado: GitHub
Models permite ~2 req/min). Además, KAOS ya persiste los resúmenes como
artifacts, por lo que un resumen previo puede reutilizarse cuando la conversación
no cambió. Los resúmenes son, a la vez, conocimiento persistente y contexto para
futuras derivaciones.

## Decision

### Cache read-through sobre Storage

- `src/kaos/core/cache.py` define `SummaryCache`, una capa **read-through** sobre
  el contrato `Storage` (no un backend nuevo). El Core sigue agnóstico: con
  `PostgresStorage` la cache sobrevive entre corridas; con `InMemoryStorage` dura
  el proceso.
- La identidad de una conversación es `(workspace, thread_id)`. El cambio de
  contenido se detecta con `content_fingerprint(message_ids)`: un SHA-256 de los
  ids de mensaje en orden. Un mensaje nuevo o borrado cambia el fingerprint.
- `SummaryCache.get(workspace, thread_id, fingerprint)` busca un artifact
  `conversation.summary` cuya metadata (`thread_id`, `content_hash`) coincida.

### Integración en el forum backfill

- Antes de llamar al LLM, `kaos backfill-forum` consulta la cache. En un **hit**
  (conversación sin cambios) reutiliza el resumen y **no llama al LLM**. En un
  **miss** resume, persiste evidencia (eventos) y el resumen (con `content_hash`
  y `thread_id` en metadata) para futuras corridas.
- `--force` ignora la cache y recomputa todos los hilos ("correr todo de nuevo").

## Consequences

- Correr el backfill repetidamente sobre temas estables cuesta cero llamadas al
  LLM; solo se recomputan los hilos con mensajes nuevos.
- Los resúmenes quedan como conocimiento persistente y trazable (cada uno
  referencia sus `source_events`), disponibles como contexto para agentes
  futuros.
- Bajo *Immutable Evidence*, `--force` agrega nuevas versiones del resumen en
  lugar de sobrescribir; la cache devuelve una versión con el mismo
  `content_hash` (misma entrada). La deduplicación / "última gana" y una cache
  incremental (que alimente el resumen siguiente con el anterior) quedan como
  evolución para la Beta.
