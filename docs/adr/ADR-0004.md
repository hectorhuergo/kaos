# ADR-0004 — CLI Scaffolding & Dogfooding

Status: Accepted

## Context

Uno de los principios de desarrollo de KAOS es el **dogfooding**: KAOS debe
terminar administrando su propio desarrollo. Hasta ahora los plugins (p. ej. el
Resume Agent) se creaban escribiendo los archivos a mano, lo que contradice ese
principio y el objetivo del EPIC-001 (KDK), que incluye "Project templates".

## Decision

El CLI incorpora un comando de scaffolding:

```
kaos new [agent|connector|publisher] <name>
```

- Las plantillas viven en `templates/<kind>/` como archivos `*.py.tmpl` con
  marcadores `__CLASS_NAME__`, `__MODULE_NAME__` y `__PLUGIN_NAME__`.
- La lógica está en `src/kaos/cli/scaffold.py`, desacoplada del parseo de
  argumentos (`src/kaos/cli/main.py`).
- La derivación de nombres normaliza kebab/snake/camelCase y evita duplicar el
  sufijo (`resume-agent` no produce `ResumeAgentAgent`).
- El comando genera el plugin **y su test**, crea el `__init__.py` del paquete
  si falta y nunca sobrescribe archivos existentes.
- El plugin generado implementa el contrato correspondiente y **no modifica el
  Core**.

## Consequences

- Los plugins se crean de forma consistente y con pruebas desde el primer
  momento, reforzando la Definition of Done.
- KAOS empieza a construirse con sus propias herramientas (dogfooding).
- Las plantillas deben mantenerse alineadas con la evolución de los contratos
  (ADR-0003). Un cambio de firma en un contrato implica actualizar la plantilla
  correspondiente.
- Limitación actual: el comando localiza `templates/` subiendo desde el paquete,
  por lo que funciona en el repositorio y en instalación editable. Empaquetar
  las plantillas como package-data para wheels distribuidos queda como trabajo
  futuro.
