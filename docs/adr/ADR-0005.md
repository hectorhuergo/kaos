# ADR-0005 — Conversation Context & LLM Provider

Status: Accepted

## Context

Al ejecutar el primer pipeline completo (`kaos up`) apareció un desajuste: el
Discord Connector emite un evento `message.created` por mensaje y el Runtime
despachaba **por evento**, construyendo un `Context` con un único evento. El
Resume Agent, que resume una **conversación completa**, se ejecutaba una vez por
mensaje y producía múltiples resúmenes.

Además, KAOS debe poder usar un proveedor de IA real sin acoplar el Core a él
(principio *AI Provider Agnostic*).

## Decision

### Motor de contexto mínimo

- El `KaosRuntime` acumula los eventos por `workspace` durante la ejecución y
  pasa a los agentes la **conversación completa** como `Context`, no solo el
  evento que dispara.
- Se introduce un evento de disparo de nivel conversación:
  `conversation.completed`.
- El Resume Agent `accepts()` solo cuando el `Context` contiene un evento
  `conversation.completed`, y resume todos los `message.*` acumulados. Así una
  conversación produce **un único** resumen.
- El Discord Connector puede emitir el trigger (`emit_completed=True`) al
  terminar de drenar un lote de mensajes.

### LLM Provider

- Se añade `OpenAICompatibleLLMProvider` (`src/kaos/plugins/providers/`), que
  implementa el contrato `LLMProvider` contra cualquier endpoint
  OpenAI-compatible (`/chat/completions`): OpenAI, Azure, **GitHub Models**,
  Ollama, etc.
- Incluye un factory `github_models(token=...)`. El uso de "Copilot" se resuelve
  apuntando a GitHub Models con un `GITHUB_TOKEN`; la API de Copilot del editor
  no está pensada para este uso.
- El `httpx.AsyncClient` es inyectable, lo que permite probar el provider sin
  red (con `httpx.MockTransport`).

## Consequences

- El pipeline conversación → resumen es coherente y produce un solo artifact.
- El motor de contexto actual es en memoria y por proceso; un Context Engine
  persistente y un Scheduler quedan para la Beta.
- KAOS puede usar proveedores reales sin que el Core dependa de ellos.
- Se incorpora `httpx` como dependencia y una nueva categoría de plugin
  (`plugins/providers/`) para implementaciones de `LLMProvider`.
